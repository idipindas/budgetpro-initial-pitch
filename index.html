STEP 01

Define the Problem Precisely

Before writing a line of code, I wrote a problem statement.
❌ What Existing Apps Do
Most budgeting apps are glorified spreadsheet viewers. They give you data, but zero insight. They tell you what happened — never why, and never what to do about it.
  • Log an expense. See it in a pie chart. Close the app.
  • No context for whether spending is "good" or "bad"
  • No ability to query your own financial data
  • No follow-up. No conversation. No intelligence.
  • Expenses tracked by date range — breaks on overlapping periods
✅ What I Set Out to Build
A system where the interface itself is the intelligence layer. Not a dashboard you read — an assistant you talk to.
  • AI with full access to live budget and expense data
  • Stateful multi-turn conversation — follow-ups just work
  • Category-level granularity with real-time limit tracking
  • Expenses stamped to budget by FK — not fragile date matching
  • PWA: native install, offline-capable, zero app store friction

// Insight: The problem wasn't "no budgeting app exists." The problem was "no budgeting app makes data queryable in natural language." That single distinction drove every architecture choice downstream.

STEP 02

Design the Solution Layer by Layer

Every technology choice was made to serve the problem — not the resume.
Frontend
React + TypeScript + Vite + TanStack React Query v5
React Query replaced the entire useEffect/useState fetch pattern. Caching, background refetch, mutation invalidation — all declarative. Navigating back to a page shows instant cached data, not a spinner.
Core UI
Styling
TailwindCSS + Lucide React
Utility-first CSS eliminates style drift. Consistent spacing tokens across components. Lucide provides a coherent icon vocabulary without bundle bloat from icon libraries.
Design
Backend
Node.js + Express + TypeScript + TypeORM + Joi
TypeORM with synchronize:true for rapid schema iteration. Joi validation middleware with a global error handler — no more silent validation failures. JWT for stateless auth.
API
Database
PostgreSQL via Neon (Serverless)
Serverless Postgres removes infrastructure overhead. Neon's branching model enables safe schema experiments. Relational model enforces budgetId FK integrity — expenses can't float between periods.
Data
AI Layer
Google Gemini 2.5 Flash — startChat() with full context injection
Each request includes the full conversation history array + current budget/expense snapshot as system context. Gemini replays context on every turn — enabling true multi-turn financial Q&A with follow-up awareness.
AI
Delivery
PWA via vite-plugin-pwa (Workbox) · Deployed on Render
beforeinstallprompt captured via custom usePWAInstall hook. Gracefully degrades when unavailable (iOS Safari, already-installed state). Service worker handles offline caching strategy.

STEP 03

Architecture Decisions — With the Tradeoffs

Good engineers don't just pick tools. They explain why — and what they gave up.
⚡
React Query over useEffect fetch loops

Four separate pages all called /budget/list. With raw fetch, that's 4 network requests on every mount. React Query deduplicates to 1, caches with a 3-minute stale time, and auto-invalidates when a mutation fires.

// Before: 4x fetch, no cache, no sync
// After: 1x fetch, shared cache,
// mutation invalidation cascades to all
🔗
budgetId FK on expenses — not date range matching

Original design matched expenses to budgets by date range overlap. When periods overlapped, expenses bled across budgets incorrectly. A direct FK relationship makes each expense's ownership unambiguous and query logic trivial.

// Bad: WHERE date BETWEEN start AND end
// Edge: overlapping budgets = wrong result
// Good: WHERE expense.budgetId = budget.id
🤖
Personality-first system instruction for AI

First prompt: "Output ONLY markdown. Do not answer outside the data." Result: Fin refused to greet users or give financial advice. Fix: define capabilities and personality first, then add constraints. Same model — completely different product.

// Lesson: Restrictions without identity
// = a broken assistant, not a safe one.
// Define who it IS before what it can't do.
🔌
PORT env var collision on cloud deploy

Render sets process.env.PORT=10000 for the web server. TypeORM config was reading the same variable for the database port. Postgres tried to connect on port 10000. Fix: renamed DB config to read DB_PORT. Systematic env var namespacing prevents this class of bug.

// Before: DB_HOST, PORT (collision)
// After: DB_HOST, DB_PORT, PORT
// Rule: namespace all env vars by service
🪝
Custom usePWAInstall hook

The native beforeinstallprompt event is ephemeral — you get one shot to capture it. Built a hook that caches the event in a ref, exposes an install() function, and tracks state so the install button gracefully hides when unavailable.

// Captures: beforeinstallprompt event
// Exposes: install(), isInstallable
// Handles: iOS Safari, already installed
🛡️
Global Joi error handler with structured logging

Validation failures were silent. Added a 4-argument Express error handler that logs the full Joi error detail and returns a structured 400 response. Debugging went from "why is the frontend getting 400?" to instant, named field errors.
